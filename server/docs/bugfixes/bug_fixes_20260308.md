# Bug Fixes & Improvements — March 8–9, 2026

This document describes two bug fixes and a set of event-logging improvements made during a troubleshooting session triggered by a student (Keyrrison Braga, user 24) who was unable to complete a card review session. The fixes address both the immediate user-facing failure and the underlying observability gap that made the failure hard to diagnose.

---

## Background: The Triggering Incident

Keyrrison reported seeing the following error message while reviewing cards in the **Verbal Tenses** deck:

> *"Missing card information in session or invalid request"*

The error appeared even after he logged out and logged back in on a different device. Because the key diagnostic log lines in `server/app.py` were at `DEBUG` level, they were invisible in the production logs, leaving no trace of what the server was actually doing when the error occurred.

---

## Part 1 — Event Logging Improvements (`server/app.py`)

### Problem

The card-fetching logic inside `get_next_card()` logged its most diagnostic lines — which queue a card was drawn from, how many new cards had been seen, and when a daily limit was reached — at `DEBUG` level. Production Gunicorn runs at `INFO` by default, so these lines were never written to the Docker log. Diagnosing review-cycle failures required either a live `DEBUG` run or manual database inspection.

Additionally, every log line that reported a card being found said, for example:

```
found learning card 1772469213105
```

with no indication of *which user* the card belonged to. In a multi-student deployment with concurrent Gunicorn workers, it was impossible to correlate a sequence of card-fetch events to a single student's session.

### Changes Made

**Five log calls promoted from `DEBUG` to `INFO`** inside `get_next_card()`:

| Location | Message |
|---|---|
| After computing day cutoff | `User {userId}, Deck {deckId}: Day Cutoff=…, Now=…, New Seen=…/{LIMIT}` |
| Learning card found | `User {userId}: found learning card {cardId}` |
| Review card found | `User {userId}: found review card {cardId} (queue=…). Card Due: …, Card Interval: … days. Current Day Cutoff: …` |
| New card found | `User {userId}: found new card {cardId}` |
| Daily new-card limit reached | `User {userId}: daily new card limit ({LIMIT}) reached for deck '{deckName}'.` |
| No new cards remaining | `User {userId}: no more new cards available in deck '{deckName}'.` |

**Key structural changes:**

- Every card-queue resolution line now carries the `User {userId}:` prefix, making per-student log correlation unambiguous even under concurrent load.
- Deck *name* (not just ID) is included in boundary messages (limit reached, no cards), so log readers do not need to look up the deck separately.

### Commit

`36afb51` — *"promote card-queue log lines from DEBUG to INFO; add User ID to all card-queue messages"*

### What the Logs Now Show (example — Keyrrison's session)

```

20:14:04  User 24: found learning card 1772469213009
20:14:10  User 24 (Keyrrison) reviewed card 1772469213009 … ease=3: Learning → Young
20:16:29  User 24: found new card 1772469213031

```

A complete picture of a student's review cycle — queue resolution, card draws, answers, and session boundaries — is now visible in the standard production log at `INFO` level.

## Part 2 — Bug Fix: Double `GET /review` on Page Load

### File

`client/src/pages/ReviewPage.jsx`

### Commit

`b580a80` — *"fix: eliminate double GET /review on ReviewPage mount"*

### Root Cause

`ReviewPage` had two `useEffect` hooks arranged in a chain:

```js
// useEffect #1 — dependency: [t]
// Reads localStorage → calls setCurrentDeckName(storedName)
useEffect(() => {
    const storedDeckName = localStorage.getItem('currentDeckName');
    if (storedDeckName) {
        setCurrentDeckName(storedDeckName);   // triggers re-render
        setReviewMessage(`…: ${storedDeckName}`);
    }
}, [t]);

// useEffect #2 — dependency: [currentDeckName, t]
// Fires whenever currentDeckName changes → calls GET /review
useEffect(() => {
    let isMounted = true;
    fetchReviewData(isMounted);             // → GET /review
    return () => { isMounted = false; };
}, [currentDeckName, t]);
```

**Execution sequence on every page load:**

1. Component mounts; `currentDeckName` state is `''` (empty initial value).
2. `useEffect #2` fires immediately → **`GET /review` call #1** sent to server.
3. `useEffect #1` fires → reads localStorage → calls `setCurrentDeckName('Verbal Tenses')`.
4. React re-renders because `currentDeckName` changed.
5. `useEffect #2` fires again → **`GET /review` call #2** sent to server.

This was confirmed in the production logs, where two consecutive `found learning card` lines appeared for the same card within one second of each other on every session start.

### Effect

Both requests returned the same card (no answer was submitted between them), so the Flask session's `currentCardId` was written twice with the same value. The user experience was correct in normal operation, but:

- One full server round-trip and one Gunicorn worker slot were wasted on every page open.
- The `isMounted` guard did not prevent the double call — the component was still mounted for both requests — creating a latent race-condition window.

### Fix

Moved the `localStorage` read to render time, before the `useState` declarations, so both `currentDeckName` and `reviewMessage` are initialized correctly on the first render. The first `useEffect` (the localStorage reader) was removed entirely.

```js
// Before
const [currentDeckName, setCurrentDeckName] = useState('');
const [reviewMessage, setReviewMessage] = useState(t('common.loading'));

// After
const storedDeckName = localStorage.getItem('currentDeckName') || '';
const [currentDeckName, setCurrentDeckName] = useState(storedDeckName);
const [reviewMessage, setReviewMessage] = useState(
    storedDeckName ? `${t('review.reviewing')}: ${storedDeckName}` : t('common.loading')
);
```

`useEffect #2` now fires exactly **once** on mount because `currentDeckName` is already correctly initialized and does not change. The server receives exactly one `GET /review` per page load.

## Part 3 — Bug Fix: Session-Expiry Error Handling

### Files

- `client/src/pages/ReviewPage.jsx`
- `client/src/localization/en/translation.json`
- `client/src/localization/pt/translation.json`
- `client/src/localization/es/translation.json`

### Commit

`5c1a5c0` — *"feat: show session-expired banner with login button on ReviewPage"*

### Root Cause

The Flask backend stores session state in the filesystem (`server/flask_session/`). Each session file has a finite lifetime. When the file expires (or is absent on a new server after a restart), the Flask session is empty. If a student attempts to submit a card answer (`POST /answer`) against an expired session, the server returns:

```
HTTP 400 — "Missing card information in session or invalid request"
```

Before this fix, the client treated this identically to any other API error: it displayed a generic **red** `alert-danger` banner with the raw server error string, and left the student on the Review page with no card and no way to recover without knowing to manually log out.

### Effect on Students

Keyrrison's case:
- Reviewed cards on his phone on the evening of March 8.
- Session expired overnight.
- Returned the next day (and tried from a second device): the new session had no `currentCardId` set because `GET /review` had not yet been called in the new session before `POST /answer` was attempted.
- Saw the cryptic error; could not review. Required instructor intervention to unblock.

### Fix

Added an `isSessionError()` helper that recognises the two server-side session-expiry signals:

```js
const isSessionError = (err) => {
    const status = err.response?.status;
    const msg = (err.response?.data?.error || '').toLowerCase();
    return status === 401 || (status === 400 && msg.includes('session'));
};
```

This check is applied in both catch blocks where the server can return such errors:

- `fetchReviewData` — `GET /review` returning 401 (authentication lost).
- `handleAnswer` — `POST /answer` returning 400 with a session-related message.

When `isSessionError` returns `true`, a `sessionExpired` boolean state is set to `true`. The JSX replaces the normal card display and error area with an **amber warning banner** containing:

1. A localized message informing the student their session has expired.
2. A **"Go to Login"** button that calls `onLogout()` (clears application state) and navigates to `/`.

### Translations

| Language | `review.sessionExpired` | `review.loginAgain` |
|---|---|---|
| English | Your session has expired. Please log in again. | Go to Login |
| Portuguese | Sua sessão expirou. Por favor, faça login novamente. | Ir para o Login |
| Spanish | Tu sesión ha expirado. Por favor, inicia sesión de nuevo. | Ir al Login |

### Recovery Flow (Post-Fix)

1. Student opens Review page with an expired session.
2. `GET /review` fails (401) **or** student submits an answer and `POST /answer` fails (400, session message).
3. Amber banner appears in the student's language with a clear explanation.
4. Student clicks **"Go to Login"** → session state cleared → redirected to login page.
5. Student logs in → `GET /review` is called → `currentCardId` set correctly → review continues normally.
